Project Proposal

The Vision

Transform the classic game of Tetris into a physical computing experiment where a small set of everyday household objects become game controllers through computer vision.

The Challenge

Within the scope of a short project, could I get a playful, working prototype where a few physical objects reliably map to core Tetris actions‚Äîwithout any cables, buttons, or controllers?

The Technology

Using a custom Teachable Machine model wrapped by ML5.js to classify live webcam frames, then translating those labels into Tetris controls inside a p5.js game engine.

Prototype Development

Stage 1: Model Training

I trained a small Teachable Machine image model with four classes: Lid, Clear Cup, Pink Cup, and Lime. Most of the work here was collecting quick samples in different positions and pruning classes that were too confusing or too similar to each other.

Stage 2: Game Engine

I built core Tetris mechanics in p5.js: piece definitions, gravity, collision detection, line clearing, scoring, and a simple speed ramp. The goal was not to recreate every Tetris rule but to make something that feels familiar and is actually playable while I fought with the ML.

function rotatePiece() {
let rotated = rotateMatrix(currentPiece.shape);
if (isValidPosition(rotated, currentPiece.x, currentPiece.y)) {
currentPiece.shape = rotated;
} else if (isValidPosition(rotated, currentPiece.x - 1, currentPiece.y)) {
currentPiece.x--;
currentPiece.shape = rotated;
} else if (isValidPosition(rotated, currentPiece.x + 1, currentPiece.y)) {
currentPiece.x++;

}

Stage 3: ML Integration

Using ML5.js, I loaded the hosted Teachable Machine model and set up a continuous classification loop. Each frame, the sketch asks the model to classify the webcam image, reads the top label and confidence, and passes them into handleTeachableControl() to trigger in-game actions.

First UI Experiments in p5.js

This project was also the first time I really played with p5.js as a UI tool‚Äînot just as a blank canvas. Instead of letting the sketch float on an empty page, I wrapped it in a simple HTML layout and treated the game like a small app.

HTML Shell Around the Sketch

The p5.js canvas lives inside a regular HTML page with a <main> container that centers everything. p5 injects the canvas there via createCanvas(), which helped me think of the sketch as one component in a larger interface, not the entire experience.

Styling the Frame & Mood

I used a full-viewport flex layout and a diagonal gradient background: background: linear-gradient(135deg, #667eea 0%, #764ba2 100%);. Paired with a monospace font and a subtle title shadow, the page feels more like a playful game frame than a raw developer demo.

Clear Instruction Panel

Below the canvas, I added a #gameInfo panel: a rounded, semi-transparent card with white text and emoji legends: ‚ÄúüéØ Lid = Rotate | ü•§ Clear Cup = Left | ü©∑ Pink Cup = Right | üçã Lime = Drop‚Äù. This was my first pass at pairing a p5 sketch with clear, friendly instructions that feel integrated into the UI.

Separating Layout & Logic

All the game logic lives in sketch.js, while the HTML + CSS handle layout and styling. That small separation taught me how to use p5.js alongside ‚Äúnormal‚Äù web UI: treat the sketch as an interactive view inside a designed frame.

Key Findings

  • Detection works best with strong contrast between object and background and a fairly uncluttered frame.
  • A confidence threshold around 0.6‚Äì0.8 plus a short cooldown keeps the game from spamming actions.
  • Lighting and camera angle matter more than the exact object‚Äîsmall setup changes can completely change performance.
  • Even with only four classes, the model still confuses objects when they overlap, move fast, or leave the center of frame.
  • Objects at roughly 1‚Äì3 feet from the camera worked best for this quick prototype.

Documented Failures

Most of the learning came from things breaking. A few patterns I kept running into:

Motion Blur

Rapid object movement causes detection loss and double triggers.

Occlusion

Hands, edges of the frame, or partial views of the object often break recognition.

Similar Objects

Very similar shapes or colors, or the same object in unexpected orientations, are frequently misclassified.

Distance Limits

Objects too far or too close to the camera fail to detect consistently.

Low Light

Poor lighting quickly degrades performance and sometimes makes the model unusable.

Background Noise

Busy backgrounds and extra objects in frame lead to false positives and jitter.

Project Reflection

Object Tetris explores the boundaries between physical and digital interaction on a realistic, small-project scale: one webcam, four objects, and a lot of trial and error. The goal wasn‚Äôt to ship a polished product‚Äîit was to get a working, end-to-end system where machine learning genuinely changes how you play.

A few key learnings: turning ML5‚Äôs detection latency into a gameplay feature that requires anticipation; using common household items to democratize the interface; and embracing the ‚Äúmessiness‚Äù of real-time classification as part of the experience instead of something to completely smooth out.

Within that constraint, this prototype points toward more accessible gaming interfaces, gesture-based controls, and augmented experiences that blend our physical and digital worlds as models and hardware continue to improve.

Next Steps: From Objects to Gestures

After building this version, my next goal is to move beyond object-based controls and explore gesture-only interaction. Right now, the system watches for four specific props: a lid, two cups, and a lime. In the next iteration, I want the camera to respond directly to how a person moves‚Äîhand positions, simple poses, or directional gestures‚Äîinstead of what they are holding.

That shift would turn Object Tetris from ‚Äúshow the right object to trigger an action‚Äù into a more embodied interface where players can rotate, shift, or drop pieces using gestures alone. It also opens up new design questions: how simple the gesture vocabulary should be, how reliably it can be recognized in real time, and how to keep the game readable and fun while the input becomes more abstract.
