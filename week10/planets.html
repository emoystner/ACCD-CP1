Cosmic Orbits: Trig, Time, and Noise in Motion

An outer-space inspired p5.js animation that turns the solar system into a lab for exploring trigonometric motion, timed events with millis(), and organic variation using randomness and Perlin noise.

The Assignment

This project was created for a creative coding homework assignment using p5.js. The goal was to design an animation that combines mathematical motion, timed events, and organic variation in a visually cohesive scene.

  • Create an animation with at least two moving shapes.
  • One shape must move in a repeated motion described by a trigonometric function (for example, using sin() and cos() to create an orbit or oscillation).
  • A second shape must perform a movement after a predetermined interval of time has passed, using a time function such as millis().
  • The two motions should be de-synchronized to create a more complex, layered rhythm instead of everything looping in sync.
  • Use randomness and/or Perlin noise to create more organic shapes or motion so the scene feels less mechanical.
  • Capture multiple iterations of the sketch and document how the work evolved over time.
  • Embed the final sketch and the iteration sketches from openProcessing or p5.js into this webpage as part of the documentation.

Note: I used ChatGPT as an assistive tool for brainstorming ideas, debugging, and polishing some of the wording on this page. All final code, design decisions, and testing were done by me.

Process & Iterations

Each version of the sketch builds on the previous one, gradually increasing the complexity of the motion, the number of celestial bodies, and the use of time and randomness.

HW V1 – First Orbit Test

In this version, I focused on getting one planet to orbit smoothly around the center using sin() and cos() for its x and y position. This was my basic test of trigonometric motion in p5.js, and it confirmed that a simple angle-plus-radius setup could create clean circular movement.

HW V2 – Details and Better Sun

Here I started building out the scene: I added a starfield background and experimented with making the sun feel more interesting than a flat circle. I also refined Earth so it had an atmosphere and simple “continents,” which made the orbiting motion visually clearer and more satisfying to watch.

HW V3 – New Sun

I didn’t love the original sun, so this version swaps it out for a noisy, multi-layered sun built from several overlapping circles. Perlin noise drives the radius and slight position offsets, making the sun “breathe” and flicker like an energetic star instead of staying perfectly static.

HW V4 – MOON

In this iteration, I added a moon that orbits Earth. The moon uses its own angle, radius, and cos()/sin() calculations, but it is drawn relative to Earth’s position. This created a nested orbit system and highlighted how trigonometric motion can be composed hierarchically.

HW V5 – SOLAR SYSTEM

This step expanded the scene into a full stylized solar system. I added all the major planets, each with different orbit radii and angular speeds. By giving each planet a unique speed, the orbits became naturally de-synchronized, creating a more complex and layered rhythm of motion.

HW V6 – MORE MOONS

In this version, I added additional moons to other planets—Jupiter, Mars, and Uranus. Each moon received its own orbit radius, speed, and starting angle. The result is a dense choreography of smaller orbits spinning at different rates, which further enhances the de-synchronized, complex rhythm of the system.

HW V7 – Final Cosmic Composition with Comet

The final version introduces a comet that appears at random intervals and travels across the scene on a curved path. Its spawn time is controlled using millis(), and each pass uses randomized starting edges, directions, and speeds. A glowing tail composed of fading circles trails behind it, creating a striking, non-looping contrast to the steady orbital motion of the planets.

How the Code Works

Under the hood, the sketch combines trigonometric functions, time-based logic, randomness, and noise to build a dynamic outer-space animation.

Here’s a high-level breakdown of the core ideas behind the animation:

Trigonometric Motion (Repeated Orbits)

Each planet’s orbit is defined using an angle and a radius. For every frame, the code updates the angle and uses cos(angle) and sin(angle) to compute the x and y position: x = cos(angle) * radius, y = sin(angle) * radius. This produces smooth circular motion. The same idea is reused for moons orbiting planets, creating nested orbital systems that all rely on trigonometric functions.

Timed Interval Movement with millis()

The comet’s movement is controlled by time. The sketch uses millis() to check how many milliseconds have passed since the program started. Once the current time passes a stored threshold (nextCometTime), the code triggers spawnComet(). The comet then moves across the screen until it exits the view, and the sketch schedules the next comet by setting a new nextCometTime further in the future.

De-Synchronizing the Motions

To avoid everything looping in perfect unison, each moving element has its own speed and sometimes its own starting phase. For example, Mercury and Neptune orbit at very different angular speeds, moons orbit at their own rates, and the comet appears based on a separate timing system. This de-synchronization creates a layered, complex rhythm that feels more like a living system than a single repeating cycle.

Randomness & Perlin Noise for Organic Feel

random() and noise() are used in several places to keep the animation from feeling too mechanical. The starfield uses randomness to scatter stars and vary their brightness. The sun uses Perlin noise to gently wobble its radius and position, giving it a flickering, breathing quality. The comet’s spawn time, starting edge, direction, and speed are randomized, so each pass through the scene feels unique and unpredictable.

Reflection

Looking back at the process of building this sketch and connecting math, time, and randomness to visual motion.

Working on this project helped me understand how powerful trigonometric functions are for animation. Using sin() and cos() to drive orbits made it clear how a simple angle and radius can turn into smooth, continuous motion. Once I had one planet working, it became almost like a pattern: I could duplicate the same logic for other planets and moons and then tweak radii and speeds to build up the solar system.

Playing with millis() and timed events shifted the way I think about rhythm in animation. Instead of everything updating in the same pattern each frame, the comet exists on its own schedule. It only appears after certain intervals have passed, and each appearance is slightly different. Combined with different orbit speeds, this made the scene feel much more alive and less like a single looping GIF.

De-synchronizing the motions and adding randomness and Perlin noise turned out to be the key for making the scene feel organic. The noisy sun, twinkling stars, and irregular comet passes keep the viewer’s eye moving. In the future, I’d like to explore adding interactivity (for example, reacting to mouse or audio input), more detailed planet textures, and maybe sound-reactive movement so the solar system could respond to music in real time.
